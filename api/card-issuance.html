Link Card to Account

POST
https://uat-api.enza.cloud/TX/v1
/cards/link-card-to-account

Link a card to an account with flexible linking modes. This operation establishes a relationship between a card and an account, enabling specific transaction capabilities.

Linking Modes:

  • Access: Card can authorize transactions on the account (authorization flow)
  • Charge: Card can be charged fees from the account
  • Access,Charge: Both authorization and charging capabilities

Validation Flow:

  1. Verify card status is "New" or "Active"
  2. Verify card contract status is "Active"
  3. Verify target account status is "Active"
  4. Execute linking operation if all validations pass

Common Use Cases:

  • Link a card to an account for transaction authorization
  • Link a card to an account for fee charging
  • Enable both authorization and fee charging on a single account

Create Virtual Prepaid Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/create-prepaid

Create a virtual prepaid card for an existing customer only.

⚠️ Important: Customer Must Exist First

This API only works for existing customers. If the customer doesn't exist, you will receive an error.

Workflow:

  1. Check if Customer Exists using POST /customers/search

    • Search by nationalId, passport, or identityId
    • If customer does NOT exist → Use POST /customers/create to create the customer first
    • If customer exists → Proceed to step 2

  2. Retrieve Authorized Product Types using GET /products/retrieve

    • Select a product from the allowedProducts list
    • Pass the product name exactly as returned

  3. Create Prepaid Card using this API (POST /cards/create-prepaid)

    • Provide the customer's document ID (nationalId, passport, or identityId)
    • Provide the productType obtained from step 2
    • Optionally enable auto-activation

Process Flow:

  1. Customer Verification: Check if customer exists using document ID
  2. Product Selection: Retrieve authorized product types via GET /products/retrieve
  3. Card Issuance: Create issuing and financial contracts, issue virtual card
  4. Card Activation (Optional): Set autoActivate: true to automatically activate the card (default: false)
  5. Notifications: Use POST /v1/notification/manage after card creation if notifications are required

Note: By default, card is NOT automatically activated unless autoActivate is set to true

Create Preproduced Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/create-preproduced

Create a card under a Preproduced Contract. This API enables banks to create multiple pre-produced cards programmatically under a preproduced contract.

Required Fields:

  • contractRid: Preproduced Contract RID (e.g., PHY-VS-PRP-CLS-PreproducedContract)
  • productRid: Token Product RID (e.g., PHY-VS-PRP-CLS-PRE-Token)

Pre-requisites:

  • Preproduced Contract must exist and be active
  • Token Product RID must exist

Error Handling:

  • Invalid Contract RID: Returns 409 with ERR_INVALID_CONTRACT
  • Invalid Product RID: Returns 409 with ERR_INVALID_PRODUCT
  • Contract Not Active: Returns 409 with ERR_CONTRACT_NOT_ACTIVE

Create Supplementary Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/create-supplementary

Create a supplementary card for an existing customer.

Workflow:

  1. Retrieve Authorized Product Types using GET /products/retrieve

    • Select a product from the allowedProducts list
    • Pass the product name exactly as returned

  2. Create Supplementary Card using this API (POST /cards/create-supplementary)

    • Provide the customer's ID (nationalId, passport, or identityId)
    • Provide the productType obtained from step 1

Process Flow:

  1. Product Selection: Retrieve authorized product types via GET /products/retrieve (must contain SUPP and API)
  2. Card Issuance: Create issuing contract, issue supplementary card
  3. Card Details: Return card ID for further operations

Note: Supplementary cards are linked to existing customer accounts and require special product types containing 'SUPP' and 'API'
