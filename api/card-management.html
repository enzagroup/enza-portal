Card Management

Endpoint:https://uat-api.enza.cloud/TX/v1

Operations for creating, activating, managing card status, and retrieving card details

Link Card to Account

POST
https://uat-api.enza.cloud/TX/v1
/cards/link-card-to-account

Link a card to an account with flexible linking modes. This operation establishes a relationship between a card and an account, enabling specific transaction capabilities.

Linking Modes:

  • Access: Card can authorize transactions on the account (authorization flow)
  • Charge: Card can be charged fees from the account
  • Access,Charge: Both authorization and charging capabilities

Validation Flow:

  1. Verify card status is "New" or "Active"
  2. Verify card contract status is "Active"
  3. Verify target account status is "Active"
  4. Execute linking operation if all validations pass

Common Use Cases:

  • Link a card to an account for transaction authorization
  • Link a card to an account for fee charging
  • Enable both authorization and fee charging on a single account

Link Card to Account Headers

  • x-api-keystringrequired

    API key for authentication

  • Content-Typestringrequired

    Content type of the request body

    Default: application/json

Link Card to Account Request Body

  • cardIdstringrequired

    Required. Card ID to link

    Example: 6426
  • accountNumberstringrequired

    Required. Account Number to link to (e.g., ACC01, ACC02)

    Example: ACC01
  • kindOfLinkingstringenumrequired

    Required. Type of linking - Access | Charge | Access,Charge

    Enum values:
    Access
    Charge
    Access,Charge
    Example: Access

Link Card to Account Responses

Card successfully linked to account

  • codestringrequired
    Example: SUCCESS_CARD_LINKED
  • paramsobjectrequired

Top Up Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/top-ups

Load funds instantly onto a card. This operation credits the specified amount to the card's available balance, enabling immediate use for transactions.

Transaction Flow:

  1. Validate card existence and status
  2. Verify source account balance
  3. Execute real-time fund transfer
  4. Update card balance immediately
  5. Return transaction confirmation with approval code

Top Up Card Headers

  • x-api-keystringrequired

    API key for authentication

  • Content-Typestringrequired

    Content type of the request body

    Default: application/json

Top Up Card Request Body

  • cardIdstring

    Required. Card ID (numeric only)

    Example: 3280
  • ccystring

    Required. Currency code (numeric only, e.g., 936 for GHS)

    Example: 936
  • amountstring

    Required. Amount to top up (decimal with up to 2 decimal places)

    Example: 100.12

Top Up Card Responses

Top-up successfully completed

  • codestringrequired
    Example: SUCCESS_TOPUP_COMPLETED
  • paramsobjectrequired
POST/cards/top-ups

Reverse Top Up Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/reverse-top-ups

Reverse a previous top-up by deducting funds from the card's available balance and returning them to the bank pool account. This operation cancels a top-up transaction.

Transaction Flow:

  1. Validate card existence and status
  2. Verify card has sufficient balance
  3. Execute real-time fund transfer from card to bank pool
  4. Update card balance immediately
  5. Return transaction confirmation with approval code

Create Virtual Prepaid Card

POST
https://uat-api.enza.cloud/TX/v1
/cards/create-prepaid

Create a virtual prepaid card for an existing customer only.

Important: Customer Must Exist First

This API only works for existing customers. If the customer doesn't exist, you will receive an error.

Workflow:

  1. Check if Customer Exists using POST /customers/search

    • Search by nationalId, passport, or identityId
    • If customer does NOT exist Use POST /customers/create to create the customer first
    • If customer exists Proceed to step 2

  2. Retrieve Authorized Product Types using GET /products/retrieve

    • Select a product from the allowedProducts list
    • Pass the product name exactly as returned

  3. Create Prepaid Card using this API (POST /cards/create-prepaid)

    • Provide the customer's document ID (nationalId, passport, or identityId)
    • Provide the productType obtained from step 2
    • Optionally enable auto-activation

Process Flow:

  1. Customer Verification: Check if customer exists using document ID
  2. Product Selection: Retrieve authorized product types via GET /products/retrieve
  3. Card Issuance: Create issuing and financial contracts, issue virtual card
  4. Card Activation (Optional): Set autoActivate: true to automatically activate the card (default: false)
  5. Notifications: Use POST /v1/notification/manage after card creation if notifications are required

Note: By default, card is NOT automatically activated unless autoActivate is set to true

Create Virtual Prepaid Card Headers

  • x-api-keystringrequired

    API key for authentication

  • Content-Typestringrequired

    Content type of the request body

    Default: application/json

Create Virtual Prepaid Card Request Body

  • productTypestringrequired

    Required. Issuing product. Obtain via GET /products/retrieve. Pass the product name exactly as returned (including API suffix).

    Example: VisaPrepaidClassicVirtualAPI
  • nationalIdstring

    Conditionally Required: National ID (required if passport and identityId not provided)

    Example: 1234567890
  • passportstring

    Conditionally Required: Passport number (required if nationalId and identityId not provided)

    Example: AB1234567
  • identityIdstring

    Conditionally Required: Identity ID (required if nationalId and passport not provided)

    Example: ID123456
  • autoActivateboolean

    Optional. Automatically activate card after creation (default: false)

    Example: false
    Default: false

Create Virtual Prepaid Card Responses

Card successfully created (and activated if autoActivate was true)

  • codestringrequired
    Example: SUCCESS_REQUEST_PROCESSED
  • paramsobjectrequired
POST/cards/create-prepaid

/cards/create-supplementary

Create a supplementary card for an existing customer.

Workflow:

  1. Retrieve Authorized Product Types using GET /products/retrieve

    • Select a product from the allowedProducts list
    • Pass the product name exactly as returned

  2. Create Supplementary Card using this API (POST /cards/create-supplementary)

    • Provide the customer's ID (nationalId, passport, or identityId)

Process Flow:

  1. Product Selection: Retrieve authorized product types via GET /products/retrieve (must contain SUPP and API)
  2. Card Issuance: Create issuing contract, issue supplementary card

Note: Supplementary cards are linked to existing customer accounts and require special product types containing 'SUPP' and 'API'
